## 問題分析

### 現狀（v1.14）
經實際測試，確定 LIKE 和 GLOB 都不 work：
```sql
-- 正常
SELECT * FROM t WHERE name = 'hello';  -- ✅ 1 row
-- 失敗
SELECT * FROM t WHERE name LIKE '%hello%';  -- ❌ 0 rows
SELECT * FROM t WHERE name GLOB '*test*';   -- ❌ 0 rows
```

`=` 正常但 LIKE/GLOB 失敗，表示：
1. Row 讀取 ✅（`=` 可以拿到 `Value::Text("hello")`）
2. 問題在 LIKE/GLOB 的 pattern 評估環節

### 懷疑方向

**方向A**：`parse_addition()` vs `parse_unary()`

LIKE pattern 用 `self.parse_addition()` 取得 pattern expression：
```rust
// src/parser/parser.rs:700
let pattern = self.parse_addition()?;
```

`parse_addition()` 會呼叫 `parse_multiplication()`，但 `'%hello%'` 只是簡單的 `Token::LitStr`，應該沒問題...

**方向B**：WHERE 篩選條件根本沒套用

或許 WHERE 條件在 planner 層就沒正確處理 LIKE/GLOB predicate？

### 需要確認的事項
1. `parse_addition()` 在遇到 `Token::LitStr` 時，確實返回 `Expr::LitStr`？
2. `eval_expr` 評估 `Expr::LitStr` 時，是否正確返回 `Value::Text`？
3. Planner 是否正確為 LIKE/GLOB 生成 filter predicate？

### 建議調查步驟
1. 在 `src/planner/executor.rs` 的 `eval_expr` 的 `Like` 分支加入 debug print
2. 或寫一個單元測試驗證 `eval_expr(LitStr("%hello%"), ...)` 的輸出
3. 追蹤 WHERE 條件從 planner 到 executor 的完整路徑

## 使用方式（修復後）

```sql
-- LIKE（% 匹配零或多個任意字元，_ 匹配單字元）
SELECT * FROM t WHERE name LIKE '%test%';
SELECT * FROM t WHERE name LIKE 'hello%';
SELECT * FROM t WHERE name NOT LIKE '%.bak';

-- GLOB（* 匹配零或多個任意字元，? 匹配單字元，區分大小寫）
SELECT * FROM t WHERE name GLOB '*test*';
SELECT * FROM t WHERE name GLOB 'test?';
SELECT * FROM t WHERE name NOT GLOB '*.bak';
```
